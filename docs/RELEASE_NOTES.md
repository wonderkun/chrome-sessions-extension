# Release notes

## 2026-04-13 — 标签绑定与会话自动注入改版

本次更新围绕「**同站多账号**」与「**标签与会话一一对应**」重做 Cookie 自动应用与打开逻辑，并补充开发需求文档。

### 亮点

- **会话绑定标签（`boundTabId`）**  
  采集当前页时，会把**活动标签的 `tabId`** 写入该会话。之后只有「这条会话 ↔ 这个标签」在**同源**前提下才会参与自动写 Cookie，不再依赖「全库只有一个会话匹配当前 URL」的旧规则，**同 URL 多账号可并存**。

- **「一标签一会话」采集**  
  同一标签再次采集时，**更新**已绑定该标签的会话（Cookie、标题、地址等），**不会**再新建第二条，避免同一 `tabId` 挂两套数据。

- **手动「打开」三段式**  
  1）优先使用 **`tabs.get(boundTabId)`**：若标签仍在且规范化 URL 与会话一致 → 聚焦并刷新/导航；  
  2）若绑定标签仍在但 **URL 已不一致** → **不**改写该标签，**新开标签**打开会话并**把绑定迁到新标签**；  
  3）绑定失效或不可用 → 按 **规范化 URL** 查找已有标签，否则 **新建**。  
  打开成功后写入/更新 `boundTabId`；并为其它会话 **去掉重复的同一 `boundTabId`**，避免自动注入命中错误会话。

- **自动注入仅在标签变为前台时**  
  仅在 **`tabs.onActivated`** 时尝试自动 `applySessionCookies`；**已移除** `tabs.onUpdated` / `loading` 上的注入。  
  含义：用户**一直留在当前标签里操作**时，不会在每次导航/刷新时再写一遍 Cookie；**从其它标签切回本标签**时会再写一次，便于对齐登录态。

- **关闭标签即解绑**  
  监听 **`tabs.onRemoved`**，关闭的标签若被某会话引用，会 **清除** 对应会话的 `boundTabId`。

### 行为与兼容性

- **旧数据**：无 `boundTabId` 的会话 **不会**再走新的自动注入；仍可通过侧栏 **「打开」** 或 **保活** 写 Cookie。新采集的会话会自动带上绑定。
- **同源判断**：自动注入要求当前标签 URL 与会话 `url` 的 **`origin`（协议 + 主机 + 端口）** 一致。
- **保活**：逻辑未改，仍按会话快照写 Cookie、请求、回写；与 `boundTabId` 无关。

### 文档与说明

- 新增/更新 **`docs/cookie-session-tab-binding.md`**：数据模型、采集/打开/自动注入/保活、验收要点。
- 更新 **`README.md`** 中与自动写 Cookie、打开标签相关的描述，与实现一致。

### 技术摘要（供维护者）

- `Session` 增加可选字段 **`boundTabId?: number`**。  
- 侧栏：`onCapture` / `onOpen` 维护绑定与去重；`openOrRefreshSession`（`src/lib/tabs.ts`）实现打开策略。  
- 后台：`autoApplyCookies.ts`（仅 `onActivated`）、`boundTabOnRemoved.ts`（`onRemoved` 清绑定）。
