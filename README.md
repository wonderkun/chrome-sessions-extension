# 账号多开助手（Chrome MV3）

在侧栏采集当前网页的 **标题、图标、地址、Cookie**，按 **分组** 管理；列表中的 **标题可双击直接编辑**（SLS 默认 `日志服务: <Logstore>` 或 project；OSS 默认 `对象存储: <region>/<bucket>`，匹配 `oss.console.aliyun.com/bucket/...`）。点击会话会先 **写回 Cookie**，再在浏览器中 **打开或复用** 已有标签（**URL 比对忽略 `#` 及之后**；无匹配则新开标签）。

## 构建

```bash
npm install
npm run build:extension
```

在 Chrome 打开 `chrome://extensions`，开启「开发者模式」，**加载已解压的扩展程序**，选择本目录下的 **`dist`**。

侧栏与阿里云浮标会跟随 **`prefers-color-scheme`**（与 Chrome「外观 / 系统主题」常见配置一致）在 **浅色 / 深色** 间自动切换。

## 使用

1. 在任意 `http(s)` 页面打开扩展图标，侧栏打开本面板。
2. 点击 **采集当前页** 保存当前活动标签的 Cookie 与元数据。
3. 点击某张卡片主体：**应用 Cookie** 后，若已有标签的 URL（去 hash）与保存的一致则 **刷新该标签**，否则 **新建标签**。
4. 用下拉框切换 **分组**；**双击**会话卡片上的标题即可改名（写入 `chrome.storage.local`）。
5. **新分组** / **管理分组** 可增删自定义分组（删除分组时，其中会话会归入「未分组」）。
6. **自动写入 Cookie**（侧栏默认勾选）：任意标签在 **开始加载**（含地址栏回车、点击链接）或 **切换为当前标签** 时，若当前 URL（去 `#`，与「打开」一致）在已保存会话里 **仅有一条** 匹配，扩展会在后台立即把该会话的 Cookie 写入浏览器，一般无需再点「打开」。**同一规范化 URL 存了多条会话时不会自动**，避免串号；纯前端改地址（`pushState`）不一定会触发检测，此时仍可用「打开」或刷新。

### 阿里云控制台自动分组

在 `*.console.aliyun.com` 下采集时，会按子域名识别产品并 **自动创建 / 选用** 分组（名称形如 `阿里云 · 百炼`），例如：

- `bailian.console.aliyun.com` → 百炼  
- `ecs.console.aliyun.com` → ECS  
- `sls.console.aliyun.com` → 日志服务 SLS  
- `oss.console.aliyun.com` → OSS  

未列入映射的子域会使用 `阿里云 · <子域大写>` 作为分组名。分组 id 以 `auto-aliyun-` 开头。

### 刷登录保活（后台请求）

保活是 **主动刷登录态**，不是「只同步你已打开的标签」：

1. 把该会话 **快照里的 Cookie** 写回浏览器（`chrome.cookies.set`）。
2. 对 **保活 URL** 发 `GET` 请求（默认=会话 URL 去 hash；可在卡片里改成控制台首页等）。扩展在拥有 `<all_urls>` 的前提下使用 `fetch` + `credentials: include` + `no-cors`，尽量带上站点 Cookie，让服务端有机会 **续期 / 刷新 Session**。
3. 再用 `chrome.cookies.getAll({ url: 会话 url })` 把 **最新 Cookie** 写回该会话快照。

**定时**：约 **每 3 小时** 对所有 **未关闭「定时刷登录」** 的会话 **顺序执行**一遍（会话之间间隔约 2.5 秒，减轻同域抢 Cookie 的冲击）。随时可在侧栏点 **立即刷登录** / **全部刷登录** 手动保活（与定时互不冲突）。

### 阿里云页面浮标

在 **`*.aliyun.com` / `aliyun.com`** 页面右下角会显示扩展 **蓝色小浮标**（云形图标）。点击后会在 **当前浏览器窗口** 打开 **侧栏**，便于采集当前页、切换会话或跳转，无需先点工具栏扩展图标。

在同一类页面上 **采集当前页** 时，会经内容脚本 **优先** 在控制台账号信息区查找带 `…_parent-info` 的 `section`（前缀 hash 会变，用 `[class*="_parent-info"]` 匹配），再读其下 **第 3 个直接子级 `p`** 内 `span > span` 的文案；若 DOM 未命中，再对整页 `innerText` 做「主账号：…」、`xxx@aliyun.com`、UID 等正则。结果写入会话卡片「主账号」一行。控制台大改版后可能读不到，属正常现象。

**限制**：

- 同一 Chrome 用户配置只有 **一套 Cookie 罐**；多条会话若域名相同，顺序保活时 **后执行的会覆盖浏览器里 Cookie**，与「多账号切换」的固有限制相同。
- 若站点 **强依赖页面 JS** 或 **CSP/风控** 拦截扩展发起的请求，仅 `fetch` 可能不够，需要自行把保活 URL 换成能触发续期的地址（例如控制台首页）。
- `no-cors` 下无法读取响应体，只要请求发出并带动 Cookie，即视为一次「刷登录」尝试。

侧栏在 `storage` 更新后会自动刷新；可看到 **上次保活时间** 与错误信息（若有）。

## 权限说明

| 权限 | 用途 |
|------|------|
| `storage` | 持久化会话与分组 |
| `cookies` | 读取 / 写入站点 Cookie |
| `tabs` | 查询标签、打开 / 刷新 |
| `sidePanel` | 侧栏 UI |
| `activeTab` | 配合当前标签采集 |
| `alarms` | 定时对会话执行「写 Cookie → fetch → 回读 Cookie」刷登录 |
| `<all_urls>` | 对任意站点读写 Cookie（敏感，仅本地使用请自行权衡） |

## 开发

```bash
npm run dev
```

侧栏页面需在构建后由扩展加载；日常可先用 `vite` 调 UI，完整能力仍以 `dist` 加载为准。

## 技术栈

Vite 6、React 18、TypeScript、Tailwind CSS v4、Lucide 图标。
